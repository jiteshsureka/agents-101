FOUNDATIONS

What Is a Claude Code Session?

CONTEXT WINDOW
Everything Claude knows about your task — your prompt, your files, the conversation so far.
TOOL ACCESS
Read, Write, Edit, Bash, Grep — all scoped to the working directory you launched Claude in.
STATE
The conversation history, any memory files, and the files on disk in that working directory.
A session is ephemeral and isolated. It knows only what you told it — and nothing about what's happening in any other terminal.
ISOLATION

Three Sessions. Three Separate Rooms.

SESSION A
Working dir: ../nopstation-email-val/
Branch: feature/email-validation
Knows: checkout A only
Memory: context A only
SESSION B
Working dir: ../nopstation-order-tests/
Branch: feature/order-tests
Knows: checkout B only
Memory: context B only
SESSION C
Working dir: ../nopstation-api-changelog/
Branch: docs/api-v2.3-changelog
Knows: checkout C only
Memory: context C only
No shared memory. No cross-talk. Session A has no idea Session B exists — and that's exactly what you want.
THE PRIMITIVE

Git Worktrees — The Essential Tool

A worktree is a second (or third) checkout of your repo into a separate directory — each on its own branch. No stashing. No checkout switching. Multiple branches active simultaneously on disk.
WITHOUT WORKTREES
One working dir. Must stash changes to switch branches. Can't run two Claude sessions on two branches simultaneously.
WITH WORKTREES
Three directories. Three branches. Three Claude sessions. All live simultaneously. No stash needed.
SHOW ME

Worktree Commands

# Create three isolated worktrees (run from main repo dir)
git worktree add ../nopstation-email-val    feature/email-validation
git worktree add ../nopstation-order-tests  feature/order-tests
git worktree add ../nopstation-api-changelog docs/api-v2.3-changelog

# Verify all three are active
git worktree list

# When you're done with a worktree
git worktree remove ../nopstation-email-val
Each path is now a fully functional repo checkout on its own branch. Open a terminal in each one and launch Claude.
SELECTION

The Parallelizability Check

Run every ticket through these four questions before you assign it to a parallel session:
  • ✓ Independent files — no overlap with other parallel tickets?
  • ✓ No shared state — different DB tables, different config, different modules?
  • ✓ No ticket-to-ticket dependency — neither task needs the other's output?
  • ✓ Each task will take more than 15 minutes of Claude time?
Pass all four → safe to run in parallel. Fail any one → resolve the dependency first, then parallelize.

THE CRAFT

Write a Focused Launch Prompt

A lazy prompt wastes tokens and goes off-track. A focused prompt is the most important thing you write per session.
Context:  nop-station checkout flow. React + TypeScript. Zod for validation.

Task:     Add email validation to the address form in checkout.
          - Validate on blur and on form submit
          - Show inline error: "Please enter a valid email address"
          - Reuse existing Zod schema: src/lib/validators.ts → emailSchema

Files:    src/components/checkout/AddressForm.tsx (primary)
          src/lib/validators.ts (read-only — do not modify)

Do NOT:  Modify auth logic, backend routes, or any file outside src/components/checkout/
Pattern: Context → Task → Files → Do NOT. Every session should start with this structure.

THE CHECK

60 Seconds to Assess a Session

  • 1. Scan the last 10 lines of output — is it moving in the right direction?
  • 2. Run a quick git diff --stat — does the file set match the task?
  • 3. Check for a blocking question — if Claude is waiting on you, answer it.
  • 4. If it's on track — leave it running and move to the next session.
Most rotations are 60 seconds. You're not reviewing the code — you're checking the heading. Deep review happens at PR time.
SHOW ME

The Context Refresh Prompt

When you return to a session after 20 minutes, don't assume it remembers where it was. Ground it fast.
# The context refresh pattern — paste this when you return to a session

Review the diff so far.
Current task:  Add email validation to CheckoutForm.tsx
Status:       You implemented blur validation. Submit validation is still missing.
Next step:    Add validation on form submit — same error message as blur.
Don't touch:  src/lib/validators.ts (read-only)
This works because it re-anchors the session without repeating the full context. Fast. Precise. Prevents drift.
WATCH OUT

When a Session Gets Stuck

SIGNS
  • • Asking a clarifying question
  • • Going in the wrong direction
  • • Modifying files it shouldn't
  • • Loops on the same step
  • • Long output, wrong result
FIXES
  • • Answer the question → continue
  • • Paste the error + "fix this only"
  • • "Stop. Revert [file]. Focus on [X]."
  • • Hard reset: new prompt with tight scope
  • • Park the session, handle later
Most stalls take 30 seconds to fix. The mistake is micromanaging healthy sessions instead of fixing the stuck ones.
EDGE CASE

Mid-Flight: Unexpected Dependency

SHAZZAD · HOUR 1 "Session B discovered it needs a validateOrder() helper that Session A is also building. They'll conflict."
OPTION 1 · PARK
Park Session B. Let A finish + merge. Then resume B on top of A's output. Safest.
OPTION 2 · SPLIT
Build the helper in a shared utility PR. Both sessions depend on it. Merge first, then both continue.
Unexpected dependencies happen. The fix is never to let both sessions race on the same code. Park one. Sequence properly.

WATCH OUT

Context Isolation Pitfalls

  • ✗
    Vague scope: "Work on the checkout form" — Claude doesn't know which files to avoid. Be explicit.
  • ✗
    Missing constraints: Forgetting "Do NOT modify validators.ts" — session edits what it shouldn't.
  • ✗
    Two sessions, same file: Both write to src/config/app.ts → guaranteed merge conflict.
  • ✓
    Fix: Always include "Files: X only" and "Do NOT touch: Y" in every launch prompt.
WATCH OUT

Git Conflicts — Rare but Real

PREVENTION
  • • Run the parallelizability check
  • • Explicit "Files: X only" per session
  • • Different modules per session
  • • No shared config/schema files
DETECTION
  • • git diff main --stat before opening PR
  • • GitHub conflict indicator on PR
  • • Check overlapping changed files
RECOVERY
  • • Rebase one branch onto the other
  • • Manual merge the conflict sections
  • • Or: ask Claude to resolve it in one session
If you ran the parallelizability check upfront, conflicts are rare. When they happen — they're small and resolvable in minutes.
COST

3 Sessions ≠ 3× Your Bill

THE MYTH
"3 sessions = 3× token cost. It'll triple our Claude bill."
THE REALITY
You pay for tokens used — not time open. Idle sessions cost nothing. Focused prompts = less wasted output tokens.
Track cost per session with /cost before closing. Tight prompts are cheaper than vague ones. Parallel efficiency offsets the extra sessions.
