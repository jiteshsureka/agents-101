THREE CONTROL LAYERSতিনটি নিয়ন্ত্রণ স্তর

The Safety Stackসেফটি স্ট্যাক

1

Permissionsপারমিশন

What agents are allowed to do — bash commands, file paths, MCP servers. Deny by default.এজেন্টরা কী করতে পারবে — bash কমান্ড, ফাইল পাথ, MCP সার্ভার। ডিফল্টে অস্বীকার।

2

Cost Controlsখরচ নিয়ন্ত্রণ

What agents are allowed to spend — per-session budgets, team caps, alerts before hard limits.এজেন্টরা কতটুকু খরচ করতে পারবে — সেশন বাজেট, টিম সীমা, হার্ড লিমিটের আগে সতর্কতা।

3

Observabilityদৃশ্যমানতা

What agents actually did — audit logs, tool traces, cost history. Can't fix what you can't see.এজেন্টরা আসলে কী করেছে — অডিট লগ, টুল ট্রেস, খরচের ইতিহাস। দেখতে না পেলে ঠিক করা যায় না।

Each layer is independent. You can add them in any order. But all three together is what "production-ready" means for team-scale agents. প্রতিটি স্তর স্বাধীন। যেকোনো ক্রমে যোগ করতে পারেন। কিন্তু তিনটি মিলে "প্রোডাকশন-রেডি" মানে টিম-স্কেল এজেন্টের জন্য।
LAYER 1: PERMISSIONSস্তর ১: পারমিশন

Deny by Default, Allow Explicitlyডিফল্টে অস্বীকার, স্পষ্টভাবে অনুমতি

✓ Allow list (add what team needs)✓ অনুমতি তালিকা
  • ✓git add, git commit, git push (non-force)git add, commit, push (নন-ফোর্স)
  • ✓npm install, npm test, npm run build
  • ✓Read/write under src/, tests/src/, tests/-এর নিচে পড়া/লেখা
  • ✓gh pr create, gh pr comment
  • ✓Read-only DB access (SELECT only)শুধু-পড়া DB অ্যাক্সেস (শুধু SELECT)
✗ Block list (always deny these)✗ ব্লক তালিকা (সবসময় অস্বীকার)
  • ✗rm -rf (any path)
  • ✗git push --force / git push origin main
  • ✗DROP TABLE, DELETE FROM (no WHERE)
  • ✗curl * | bash (pipe to shell)
  • ✗Any write to infrastructure/, .envinfrastructure/, .env-এ লেখা
Config lives in CLAUDE.md (project-level) and .claude/settings.json. Every session inherits it automatically. কনফিগ CLAUDE.md (প্রজেক্ট-লেভেল) এবং .claude/settings.json-এ থাকে। প্রতিটি সেশন স্বয়ংক্রিয়ভাবে এটি উত্তরাধিকার সূত্রে পায়।
BASH CONTROLSBASH নিয়ন্ত্রণ

Blocking Dangerous Commandsবিপজ্জনক কমান্ড ব্লক করা

# .claude/settings.json — bash command restrictions
{
  "permissions": {
    "allow": [
      "Bash(git add:*)",
      "Bash(git commit:*)",
      "Bash(git push origin feat/*:*)",  // feature branches only
      "Bash(npm *:*)",
      "Bash(gh pr *:*)"
    ],
    "deny": [
      "Bash(git push --force*)",
      "Bash(git push origin main*)",
      "Bash(rm -rf*)",
      "Bash(curl*|*bash*)"
    ]
  }
}
The deny list is enforced before Claude even tries. An agent cannot bypass it with clever prompting — it's a hard system-level block, not an instruction. ক্লড চেষ্টা করার আগেই ডেনি লিস্ট কার্যকর হয়। একটি এজেন্ট চালাক প্রম্পটিং দিয়ে এটি বাইপাস করতে পারে না — এটি একটি হার্ড সিস্টেম-লেভেল ব্লক, নির্দেশ নয়।
APPROVAL GATESঅনুমোদন গেট

Human-in-the-Loop for Irreversible Actionsঅপরিবর্তনীয় অ্যাকশনের জন্য লুপে মানুষ

Some actions should need explicit human approval — not just a permission check. Hook exit code 2 pauses Claude and surfaces the decision to a human before continuing. কিছু অ্যাকশনের জন্য শুধু পারমিশন চেক নয়, স্পষ্ট মানবিক অনুমোদন দরকার। হুক এক্সিট কোড 2 ক্লডকে থামায় এবং চালিয়ে যাওয়ার আগে মানুষের সামনে সিদ্ধান্তটি উপস্থাপন করে। 
#!/bin/bash
# scripts/prod-guard.sh — PreToolUse:Bash hook

INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command')

# Flag anything touching production
if echo "$COMMAND" | grep -qiE '(prod|production|--force|DROP|DELETE.*FROM)'; then
  echo "⚠️  Production-affecting command detected." >&2
  echo "Command: $COMMAND" >&2
  echo "Pausing for human review before proceeding." >&2
  exit 2   # exit 2 = ask human, don't block entirely
fi

exit 0
exit 1 blocks silently. exit 2 blocks and asks the human to approve or reject before Claude continues. Use exit 2 for irreversible actions. exit 1 চুপচাপ ব্লক করে। exit 2 ব্লক করে এবং ক্লড চালিয়ে যাওয়ার আগে মানুষকে অনুমোদন বা প্রত্যাখ্যান করতে বলে।
LAYER 2: COST CONTROLSস্তর ২: খরচ নিয়ন্ত্রণ

COST ALERTSখরচ সতর্কতা

Hook on Agent Spawnএজেন্ট স্পনে হুক

#!/bin/bash
# scripts/cost-guard.sh — PreToolUse:Agent hook

INPUT=$(cat)
# Read today's spend from your tracking file
TODAY_SPEND=$(cat .claude/spend-today.json | jq '.total')
DAILY_LIMIT="15.00"
WARN_THRESHOLD="12.00"

if (( $(echo "$TODAY_SPEND >= $DAILY_LIMIT" | bc) )); then
  echo "Daily limit $DAILY_LIMIT reached. Agent spawn blocked." >&2
  exit 1
fi

if (( $(echo "$TODAY_SPEND >= $WARN_THRESHOLD" | bc) )); then
  echo "⚠️ Spend at $TODAY_SPEND — approaching daily limit." >&2
  # exit 0 — warn but allow
fi

exit 0
Track spend per engineer per day. Aggregate to team total. Alert at 80%, hard block at 100%. Update .claude/spend-today.json from PostToolUse hook on every Agent call. প্রতিদিন প্রতি ইঞ্জিনিয়ারের খরচ ট্র্যাক করুন। টিম মোটে একত্রিত করুন। ৮০%-এ সতর্ক, ১০০%-এ হার্ড ব্লক।

INCIDENT FIXইনসিডেন্ট সমাধান

Three Changes That Prevent Itতিনটি পরিবর্তন যা এটি প্রতিরোধ করে

Fix 1: Depth limit in promptসমাধান ১: প্রম্পটে ডেপথ লিমিট
Every subagent prompt ends with: "Do NOT spawn any subagents. You are a worker, not an orchestrator." প্রতিটি সাবএজেন্ট প্রম্পটের শেষে: "কোনো সাবএজেন্ট স্পন করবেন না। আপনি একজন কর্মী, অর্কেস্ট্রেটর নন।"
Fix 2: Daily spend hookসমাধান ২: দৈনিক খরচ হুক
PreToolUse on Agent: check daily spend. Block if over $15 per engineer per day. Agent-এ PreToolUse: দৈনিক খরচ চেক করুন। প্রতিদিন প্রতি ইঞ্জিনিয়ার $১৫-এর বেশি হলে ব্লক করুন।
Fix 3: Team CLAUDE.md ruleসমাধান ৩: টিম CLAUDE.md নিয়ম
CLAUDE.md: "Max subagent depth = 1. Orchestrators spawn workers; workers return results only." CLAUDE.md: "সর্বোচ্চ সাবএজেন্ট ডেপথ = ১। অর্কেস্ট্রেটর কর্মী স্পন করে; কর্মীরা শুধু ফলাফল ফেরত দেয়।"
With all three: a 12-agent recursive chain is impossible. The same scenario today would cost ~$28 instead of $340 and Claude would stop at depth 1. তিনটি সহ: ১২-এজেন্ট রিকার্সিভ চেইন অসম্ভব। একই পরিস্থিতিতে আজ $৩৪০-এর পরিবর্তে ~$২৮ খরচ হবে এবং ক্লড ডেপথ ১-এ থামবে।

THE AUDIT HOOKঅডিট হুক

Logging Every Agent Actionপ্রতিটি এজেন্ট অ্যাকশন লগ করা

#!/bin/bash
# scripts/audit-logger.sh — PostToolUse hook (all tools)

INPUT=$(cat)
TIMESTAMP=$(date -u '+%Y-%m-%dT%H:%M:%SZ')
TOOL=$(echo "$INPUT"    | jq -r '.tool_name')
USER=${GIT_AUTHOR_NAME:-"unknown"}

# Build audit record
RECORD=$(echo "$INPUT" | jq --arg ts "$TIMESTAMP" --arg u "$USER" '{
  timestamp: $ts,
  user: $u,
  tool: .tool_name,
  input: .tool_input,
  exit_code: .tool_response.exit_code
}')

# Append to daily audit log
echo "$RECORD" >> .claude/audit-$(date +%Y-%m-%d).jsonl

exit 0
One JSONL line per tool call. Every session, every engineer. Searchable with jq. Tells you exactly what happened and when — months later. প্রতিটি টুল কলে একটি JSONL লাইন। প্রতিটি সেশন, প্রতিটি ইঞ্জিনিয়ার। jq দিয়ে অনুসন্ধানযোগ্য। মাস পরেও ঠিক কী ঘটেছিল বলে।

CLAUDE.md is the one file every agent session reads at startup. Make it your team's safety rules, not just project conventions. CLAUDE.md হলো একমাত্র ফাইল যা প্রতিটি এজেন্ট সেশন স্টার্টআপে পড়ে। এটিকে শুধু প্রজেক্ট কনভেনশন নয়, আপনার টিমের সেফটি নিয়মে পরিণত করুন। 
# CLAUDE.md — Team Safety Section (add to every project)

## Agent Safety Rules (mandatory)

**Subagent depth limit:** Max 1 level. You may spawn workers.
Workers may NOT spawn further agents. State this explicitly
in every subagent prompt you write.

**Production rule:** Never write to production databases,
never force-push to main, never modify infrastructure/.
These require a separate human-approved PR.

**Cost rule:** If a task would require more than 5 subagents
or read more than 50 files, stop and ask the engineer first.

**Credentials rule:** Never include API keys, DB passwords,
or secrets in your prompts or output files. They will be
logged in the audit trail.
SAFE ROLLOUTনিরাপদ রোলআউট

Three Phases, Not One Big Bangতিনটি পর্যায়, এক বিগ ব্যাং নয়

Phase 1 — Pilotপর্যায় ১ — পাইলট
2 engineers, 1 week. Tight permissions. Daily spend review. Collect edge cases and failure modes. ২ ইঞ্জিনিয়ার, ১ সপ্তাহ। কঠোর পারমিশন। দৈনিক খরচ রিভিউ। এজ কেস এবং ফেলিয়ার মোড সংগ্রহ।
Phase 2 — Monitoredপর্যায় ২ — পর্যবেক্ষিত
Full team, safety stack on. Weekly audit log review. Refine rules based on real usage patterns. পুরো টিম, সেফটি স্ট্যাক চালু। সাপ্তাহিক অডিট লগ রিভিউ। বাস্তব ব্যবহারের প্যাটার্নের উপর ভিত্তি করে নিয়ম পরিমার্জন।
Phase 3 — Autonomousপর্যায় ৩ — স্বায়ত্তশাসিত
Automated monitoring. Alerts instead of manual checks. Team self-sufficient with established norms. স্বয়ংক্রিয় মনিটরিং। ম্যানুয়াল চেকের পরিবর্তে সতর্কতা। প্রতিষ্ঠিত নিয়ম সহ টিম স্বনির্ভর।
Rifat's mistake: skipping Phase 1 and going straight to Phase 3. Two engineers for one week would have caught the recursive spawn problem for $40, not $340. রিফাতের ভুল: পর্যায় ১ বাদ দিয়ে সরাসরি পর্যায় ৩-এ যাওয়া। দুই ইঞ্জিনিয়ার এক সপ্তাহের জন্য $৩৪০ নয়, $৪০-এ রিকার্সিভ স্পন সমস্যা ধরত।

Credentials in Promptsপ্রম্পটে ক্রেডেনশিয়াল

Claude logs context. Your audit hook logs tool inputs. If an engineer puts a database password in a task description — "connect to postgres://user:p4ssw0rd@prod:5432/db" — that string is now in your audit logs, your session history, and potentially your context window cache. ক্লড কনটেক্সট লগ করে। আপনার অডিট হুক টুল ইনপুট লগ করে। কোনো ইঞ্জিনিয়ার টাস্ক বিবরণে ডেটাবেজ পাসওয়ার্ড রাখলে — সেই স্ট্রিং এখন আপনার অডিট লগ, সেশন ইতিহাস, এবং সম্ভবত কনটেক্সট ক্যাশে আছে।
✓ Do this✓ এটা করুন
Store credentials in env vars or secrets managers. Reference them in prompts as $DB_URL. The value never appears in the prompt text. ক্রেডেনশিয়াল env ভেরিয়েবল বা সিক্রেটস ম্যানেজারে সংরক্ষণ করুন। প্রম্পটে $DB_URL হিসেবে রেফার করুন।
✗ Never this✗ কখনো এটা না
Inline secrets: API keys, passwords, tokens in task descriptions, CLAUDE.md, or shared-state files. ইনলাইন সিক্রেট: টাস্ক বিবরণ, CLAUDE.md, বা শেয়ার্ড-স্টেট ফাইলে API কী, পাসওয়ার্ড, টোকেন।
Add to CLAUDE.md: "Never include credentials in prompts, task descriptions, or output files. Use environment variables." CLAUDE.md-এ যোগ করুন: "প্রম্পট, টাস্ক বিবরণ, বা আউটপুট ফাইলে কখনো ক্রেডেনশিয়াল অন্তর্ভুক্ত করবেন না।"
