THE UPGRADEআপগ্রেড

From Reactive to Proactiveরিঅ্যাক্টিভ থেকে প্রোঅ্যাক্টিভ

Reactive (Eps 1–4)রিঅ্যাক্টিভ (এপি ১–৪)
You open a terminal. You type a prompt. You wait. You come back later. You trigger everything. তুমি টার্মিনাল খোলো। প্রম্পট টাইপ করো। অপেক্ষা করো। পরে ফিরে আসো। তুমিই সব ট্রিগার করো।
Proactive (Ep 5)প্রোঅ্যাক্টিভ (এপি ৫)
An event fires. Or a clock ticks. Claude wakes up, does the work, and puts the output where you need it. Nobody triggered anything. একটি ইভেন্ট ঘটে। বা একটি ঘড়ি টিক করে। ক্লড জেগে ওঠে, কাজ করে, এবং যেখানে দরকার সেখানে আউটপুট রাখে। কেউ কিছু ট্রিগার করেনি।
This is the jump from tool to teammate. A tool waits for you. A teammate shows up when the work is ready. এটাই টুল থেকে টিমমেট-এ লাফ দেওয়া। টুল তোমার জন্য অপেক্ষা করে। টিমমেট কাজ প্রস্তুত হলে নিজেই আসে।
TWO PRIMITIVESদুটি প্রিমিটিভ

Hooks vs Scheduled Agentsহুক বনাম শিডিউলড এজেন্ট

⚡ Hooks⚡ হুক
Trigger: Claude Code events (tool use, file edit, session end)
Speed: milliseconds — same session
Purpose: validate, block, log, react ট্রিগার: ক্লড কোড ইভেন্ট (টুল ব্যবহার, ফাইল এডিট, সেশন শেষ)
গতি: মিলিসেকেন্ড — একই সেশন
উদ্দেশ্য: যাচাই, ব্লক, লগ, প্রতিক্রিয়া
🕐 Scheduled Agents🕐 শিডিউলড এজেন্ট
Trigger: time (cron) or external event (CI, PR, webhook)
Speed: minutes — new Claude session
Purpose: automate recurring tasks end-to-end ট্রিগার: সময় (cron) বা বাহ্যিক ইভেন্ট (CI, PR, ওয়েবহুক)
গতি: মিনিট — নতুন ক্লড সেশন
উদ্দেশ্য: পুনরাবৃত্তিমূলক কাজ এন্ড-টু-এন্ড স্বয়ংক্রিয় করা
Hooks guard individual actions in real time. Scheduled agents run whole workflows on their own schedule. Both can run without a human present. হুক রিয়েল-টাইমে পৃথক অ্যাকশন রক্ষা করে। শিডিউলড এজেন্ট নিজের শিডিউলে পুরো ওয়ার্কফ্লো চালায়। দুটোই মানুষ ছাড়া চলতে পারে।
HOOKS ANATOMYহুকের গঠন

How Hooks Workহুক কীভাবে কাজ করে

A hook is a shell command wired to a Claude Code event. Claude fires the event → your shell script runs → exit code tells Claude what to do next. হুক হলো একটি ক্লড কোড ইভেন্টের সাথে সংযুক্ত শেল কমান্ড। ক্লড ইভেন্ট ফায়ার করে → আপনার শেল স্ক্রিপ্ট চলে → এক্সিট কোড ক্লডকে জানায় পরবর্তীতে কী করতে হবে।
PreToolUse — before any tool runsPreToolUse — যেকোনো টুল চলার আগে
PostToolUse — after a tool completesPostToolUse — টুল শেষ হওয়ার পরে
Stop — when Claude finishes its turnStop — ক্লড তার টার্ন শেষ করলে
Notification — when Claude needs attentionNotification — ক্লডের মনোযোগ দরকার হলে
# .claude/settings.json — hook config
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{ "type": "command", "command": "./scripts/check-bash.sh" }]
    }]
  }
}

# Exit codes:
exit 0  → allow Claude to proceed
exit 1  → block + show stderr to Claude as context
exit 2  → block + ask human before continuing
HOOK USE CASESহুকের ব্যবহার

What Hooks Are Good Forহুক কীসের জন্য ভালো

  • 🔒PreToolUse: block dangerous bash commands before they run (rm -rf, force push)PreToolUse: বিপজ্জনক bash কমান্ড চলার আগে ব্লক করুন (rm -rf, ফোর্স পুশ)
  • ⚡PostToolUse: run type-check after every file edit — Claude sees failures immediatelyPostToolUse: প্রতিটি ফাইল এডিটের পরে টাইপ-চেক চালান — ক্লড সাথে সাথে ব্যর্থতা দেখে
  • 💰PreToolUse: log token estimate before subagent spawn — catch runaway costs earlyPreToolUse: সাবএজেন্ট স্পন করার আগে টোকেন এস্টিমেট লগ করুন — আগে খরচ ধরুন
  • 📋PostToolUse: append audit log entry — timestamp, tool, args — for every actionPostToolUse: প্রতিটি অ্যাকশনের জন্য অডিট লগ এন্ট্রি যোগ করুন
  • 🔔Notification: send Slack message when Claude is waiting for human inputNotification: ক্লড মানুষের ইনপুটের জন্য অপেক্ষা করলে স্ল্যাক মেসেজ পাঠান
SHOW MEদেখাও

Hook: Block Push if Tests Failহুক: টেস্ট ফেল হলে পুশ ব্লক করুন

#!/bin/bash
# scripts/pre-push-guard.sh — wired to PreToolUse:Bash

# Only intercept git push commands
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command')

if echo "$COMMAND" | grep -q "git push"; then
  # Run the test suite first
  if ! npm test --silent 2>/dev/null; then
    echo "Tests failing — push blocked. Fix tests first." >&2
    exit 1   # Claude sees this message and stops
  fi
fi

exit 0  # Tests pass — allow push
Claude tries to run git push. Hook intercepts → runs tests → tests fail → hook exits 1 → Claude sees "Tests failing — push blocked" and stops itself. Nobody had to babysit. ক্লড git push চালানোর চেষ্টা করে। হুক ইন্টারসেপ্ট করে → টেস্ট চালায় → টেস্ট ফেল → হুক 1 দিয়ে বের হয় → ক্লড "Tests failing — push blocked" দেখে নিজেই থামে।

Hook: Auto-Log Every File Editহুক: প্রতিটি ফাইল এডিট স্বয়ংক্রিয় লগ করুন

#!/bin/bash
# scripts/edit-logger.sh — wired to PostToolUse:Edit

# Append to the sprint edit log

SCHEDULED AGENTSশিডিউলড এজেন্ট

GitHub Actions + Claude Codeগিটহাব অ্যাকশন + ক্লড কোড

GitHub Actions is the most common scheduler. Claude Code is the worker. The pattern: a cron trigger fires → Actions checks out the repo → runs claude -p "your prompt" → output goes where you route it. গিটহাব অ্যাকশন সবচেয়ে সাধারণ শিডিউলার। ক্লড কোড হলো কর্মী। প্যাটার্ন: cron ট্রিগার ফায়ার → Actions রেপো চেকআউট করে → claude -p "আপনার প্রম্পট" চালায় → আউটপুট রাউট করে।

ANATOMYগঠন

What Every Scheduled Workflow Needsপ্রতিটি শিডিউলড ওয়ার্কফ্লোর যা দরকার

1. Trigger
When does it run? Cron expression, PR event, CI signal, webhook.কখন চলবে? Cron এক্সপ্রেশন, PR ইভেন্ট, CI সিগনাল, ওয়েবহুক।
2. Context
What does Claude need to know? Repo state, relevant files, last-run output.ক্লডের কী জানা দরকার? রেপো স্টেট, প্রাসঙ্গিক ফাইল, শেষ-রানের আউটপুট।
3. Task
A clear, scoped prompt. What to do. What to produce. What format.স্পষ্ট, সীমাবদ্ধ প্রম্পট। কী করবে। কী তৈরি করবে। কী ফরম্যাটে।
4. Output
Where does the result go? File commit, PR comment, Slack post, email.ফলাফল কোথায় যাবে? ফাইল কমিট, PR কমেন্ট, স্ল্যাক পোস্ট, ইমেইল।
5. Failure
What happens if it fails? Notification hook, retry policy, fallback output.ব্যর্থ হলে কী হবে? নোটিফিকেশন হুক, রিট্রাই পলিসি, ফলব্যাক আউটপুট।

The GitHub Actions Workflowগিটহাব অ্যাকশন ওয়ার্কফ্লো

# .github/workflows/release-prep.yml
name: Release Prep Agent

on:
  schedule:
    - cron: '0 23 * * 3'  # every Wednesday at 23:00 UTC

jobs:
  prep:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }  # need full history for git log

      - name: Run release prep agent
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          # Feed context + run Claude
          git log $(git describe --tags --abbrev=0)..HEAD \
            --oneline > /tmp/commits.txt
          npm test 2>&1 | tail -50 > /tmp/test-summary.txt

          claude -p ".claude/prompts/release-prep.md" \
            --no-interactive

The Release Prep Promptরিলিজ প্রেপ প্রম্পট

📄 .claude/prompts/release-prep.md
You are preparing the release notes for our Thursday ship.

Context files to read:
- /tmp/commits.txt — all commits since last release
- /tmp/test-summary.txt — current failing tests
- CHANGELOG.md — existing format to follow

Deliverables:
1. Add new changelog section to CHANGELOG.md (keep existing format)
2. Write RELEASE-NOTES.md with: summary, breaking changes, failing tests that block ship, and files changed in 2+ open PRs (risk flag)
3. Run: gh pr comment <latest-pr> --body "$(cat RELEASE-NOTES.md)"

Do NOT push to main. Do NOT merge any PRs. Read-only except CHANGELOG.md and RELEASE-NOTES.md.
Scope + deliverables + output format + explicit constraints. A scheduled agent has no human to ask — the prompt must be self-contained. স্কোপ + ডেলিভারেবল + আউটপুট ফরম্যাট + স্পষ্ট সীমাবদ্ধতা। শিডিউলড এজেন্টের জিজ্ঞেস করার মানুষ নেই — প্রম্পট অবশ্যই স্বয়ংসম্পূর্ণ হতে হবে।

CHAININGচেইনিং

COST MODELখরচের মডেল

Budget Before You Scheduleশিডিউল করার আগে বাজেট করুন

✓ Low cost runs✓ কম খরচের রান
Read a few files, write one document. ~20–40k tokens. Runs once a week. Cheap. কয়েকটি ফাইল পড়া, একটি ডকুমেন্ট লেখা। ~২০–৪০k টোকেন। সপ্তাহে একবার চলে। সস্তা।
✗ Dangerous patterns✗ বিপজ্জনক প্যাটার্ন
Runs every 5 minutes. Reads entire codebase. Spawns subagents. That's real money, fast. প্রতি ৫ মিনিটে চলে। পুরো কোডবেজ পড়ে। সাবএজেন্ট স্পন করে। এটা দ্রুত প্রচুর অর্থ।
Rule: estimate tokens per run × runs per month before scheduling. Add a --max-tokens flag. Add a notification hook on failure. Scheduled agents fail silently at 3am if you don't. নিয়ম: শিডিউল করার আগে প্রতি রানে টোকেন × মাসে রানের সংখ্যা এস্টিমেট করুন। --max-tokens ফ্ল্যাগ যোগ করুন। ব্যর্থতায় নোটিফিকেশন হুক যোগ করুন।
Also: add an idempotency check — skip the run if nothing changed since last time. Most cron triggers fire even on quiet nights. এছাড়া: আইডেম্পোটেন্সি চেক যোগ করুন — শেষবারের পর কিছু না বদলালে রান স্কিপ করুন।

Stale Contextপুরনো কনটেক্সট

A scheduled agent doesn't know what your team decided in Monday's standup. It only knows what's in the repo at run time. If your CLAUDE.md and shared-state files are outdated, the agent will act on outdated assumptions. একটি শিডিউলড এজেন্ট জানে না সোমবারের স্ট্যান্ডআপে আপনার টিম কী সিদ্ধান্ত নিয়েছে। সে শুধু রান টাইমে রেপোতে যা আছে তাই জানে। CLAUDE.md এবং শেয়ার্ড-স্টেট ফাইল পুরনো হলে এজেন্ট পুরনো অনুমানে কাজ করবে।
Fix 1: Explicit context filesসমাধান ১: স্পষ্ট কনটেক্সট ফাইল
Prompt: "Read .claude/shared-state.md and CLAUDE.md before starting. Treat them as ground truth." প্রম্পট: "শুরু করার আগে .claude/shared-state.md এবং CLAUDE.md পড়ুন। এগুলোকে সত্য হিসেবে বিবেচনা করুন।"
Fix 2: Scope narrowসমাধান ২: সংকীর্ণ স্কোপ
Scheduled agents work best on well-defined, stable tasks. Avoid tasks that need fresh human context every run. শিডিউলড এজেন্ট সু-সংজ্ঞায়িত, স্থিতিশীল কাজে সবচেয়ে ভালো কাজ করে। প্রতিটি রানে তাজা মানবিক কনটেক্সট প্রয়োজন এমন কাজ এড়িয়ে চলুন।

Silent Failuresনীরব ব্যর্থতা

Your release prep agent failed at 2am with a rate limit error. You find out Thursday morning when the changelog isn't there. Without a failure notification, scheduled agents fail quietly. রিলিজ প্রেপ এজেন্ট রাত ২টায় রেট লিমিট এররে ব্যর্থ হয়েছে। বৃহস্পতিবার সকালে জানতে পারলেন যখন চেঞ্জলগ নেই। ফেলিয়ার নোটিফিকেশন ছাড়া শিডিউলড এজেন্ট নীরবে ব্যর্থ হয়। 
# Always wrap scheduled runs with failure notification
- name: Run release prep agent
  run: |
    claude -p .claude/prompts/release-prep.md --no-interactive
  id: agent-run

- name: Notify on failure
  if: failure()
  run: |
    curl -X POST "$SLACK_WEBHOOK" \
      -d '{"text":"⚠️ Release prep agent failed. Check Actions logs."}'
Rule: every scheduled workflow must have a failure path. If it fails silently, it doesn't exist. নিয়ম: প্রতিটি শিডিউলড ওয়ার্কফ্লোতে অবশ্যই একটি ব্যর্থতার পথ থাকতে হবে।
